# PR: Advanced Chat Features

## Summary
Цей PR додає розширені функції для покращення інтерактивності, швидкості роботи та безпеки чату в реальному часі (Advanced Chat Features). Усі нові функції супроводжуються надійними перевірками прав доступу (RLS), безпечним авторизованим викликом серверних функцій без розкриття секретів та автоматичним очищенням ресурсів у фронтенді.

## Features
* **Typing indicators (Індикатори введення)** — реалізовано через Supabase Realtime Broadcast. Відображає, хто саме зараз друкує повідомлення, з оптимізованим 2-секундним дебаунсом.
* **Read receipts (Звіти про прочитання)** — реалізовано компактне відображення подвійних галочок на основі оновлення позначки `last_read_at` учасників розмови у реальному часі.
* **Voice messages (Голосові повідомлення)** — можливість записувати аудіо в інтерфейсі, автоматично конвертувати в сумісний PCM WAV формат, транскрибувати в текст через інтеграцію Dify STT та зберігати записи в захищеному сховищі.
* **Thread replies (Панель обговорень / Треди)** — можливість створювати окремі гілки коментарів до повідомлень (`parent_id`). Тред-відповіді не засмічують головний екран чату.
* **Global message search (Пошук повідомлень)** — швидкий повнотекстовий пошук (FTS) з фільтрами за датою та кімнатою розмови.
* **Push notifications (Push-сповіщення)** — надсилання push-повідомлень користувачам при появі нових повідомлень за допомогою Postgres-тригера та Edge Function `push-send`.

## Security & Stability
* **Forward-only migrations** — історичні файли міграцій не змінювалися в Git, що виключає ризик помилок `checksum mismatch` під час CI/CD збірки на продакшені.
* **Storage RLS for voice messages** — доступ до бакету `voice-messages` на запис обмежений виключно авторизованими користувачами, які є безпосередніми учасниками відповідної розмови.
* **Hardened push-send system auth** — функція `push-send` безпечно приймає запити за допомогою авторизації `service_role_key` від тригера БД, а локальні ключі обходу JWT автоматично блокуються на продакшені. CORS обмежено дозволеними доменами.
* **Search restricted to participant conversations** — клієнтський пошуковий запит примусово обмежується ідентифікаторами чатів, де поточний користувач є учасником.
* **Realtime cleanup** — усі канали та підписки Supabase Realtime гарантовано відписуються при виході з чату або розмонтуванні компонентів для запобігання витоку пам'яті.
* **Build/typecheck verified** — збірка (`npm run build`) та перевірка типів (`tsc --noEmit`) проходять зі 100% успіхом.

## Database changes
* **Нові стовпці**:
  * `public.messages.parent_id` (UUID, посилання на `public.messages(id)` з `ON DELETE CASCADE`).
* **Нові індекси**:
  * `idx_messages_parent_id` на стовпець `parent_id` для оптимізації тредів.
  * `idx_messages_content_fts` (GIN-індекс на `to_tsvector('simple', content)`) для швидкого повнотекстового пошуку.
* **Розширення**:
  * `pg_net` (для асинхронних HTTP-запитів із БД).
* **Сховище (Storage Buckets)**:
  * Бакет `voice-messages` (публічний на SELECT, захищений RLS на INSERT).
* **Політики RLS**:
  * SELECT політика для бакету `voice-messages`.
  * INSERT політика для бакету `voice-messages` (перевірка `(storage.foldername(name))[1]::uuid` на участь у чаті).
* **Тригери та функції**:
  * Функція `public.notify_message_participants()` — формує пуш-пейлоад та ініціює запит до Edge Function `push-send`.
  * Тригер `trigger_notify_message_participants` на таблиці `messages` (викликається `AFTER INSERT`).

## Edge Functions
* **`push-send`**:
  * Отримує параметри сповіщення, вилучає підписки пристроїв отримувача з `push_subscriptions`.
  * Підтримує CORS через спільний утилітарний модуль `../_shared/cors.ts` (заборона credentialed requests на `*`).
  * Здійснює внутрішню перевірку токенів при `verify_jwt = false`.
* **`dify-client`**:
  * Оновлено Zod схему для прийому метаданих голосових повідомлень (`messageType`, `fileUrl`, `transcription`, `voiceDuration`).
  * Усі запити на додавання повідомлень виконуються через клієнтський токен JWT (`userSupabase`), тому вони повністю підпорядковані загальним правилам RLS бази даних.

## Manual QA Checklist
- [ ] **typing indicator in two tabs** — перевірити відображення статусу введення тексту в реальному часі між двома різними вкладками.
- [ ] **read receipts between two users** — перевірити появу блакитних подвійних галочок при прочитанні повідомлень іншим учасником.
- [ ] **voice record/upload/playback/transcription** — записати голосове повідомлення, перевірити завантаження в бакет, успішність транскрипції та програвання плеєром.
- [ ] **thread reply panel** — написати коментар у гілку тредів, перевірити, що він показується тільки в ThreadPanel і не дублюється в головному чаті.
- [ ] **global search filters** — протестувати повнотекстовий пошук за текстом та перевірити роботу фільтрів за чатами й датами.
- [ ] **push notification subscribe/send** — підписатися на пуші у фронтенді та перевірити надходження пуш-сповіщення при отриманні повідомлення від іншого акаунта.
- [ ] **permission denied push case** — відхилити дозвіл на сповіщення в браузері та переконатися, що клієнтська частина обробляє це без збоїв (показує тост-попередження).
- [ ] **unauthorized chat access attempt** — симулювати спробу завантаження/пошуку повідомлень з чату, де користувач не є учасником, і переконатися в блокуванні запиту на рівні RLS бази даних та фільтрів.

## Rollback Plan
* Скасувати комміти цього PR (усі зміни локалізовані).
* За потреби вимкнути асинхронні запити тригера: `DROP TRIGGER IF EXISTS trigger_notify_message_participants ON public.messages;`.
* Тимчасово вимкнутиEdge-функцію `push-send` шляхом видалення або зміни змінної оточення `INTERNAL_API_KEY`.
* Усі міграції є forward-only, тому історичні дані та схема міграцій на продакшені не постраждають при скасуванні.

## Environment variables
Необхідні змінні оточення для запуску Edge Functions:
* `VAPID_PUBLIC_KEY` — публічний ключ VAPID сповіщень.
* `VAPID_PRIVATE_KEY` — приватний ключ VAPID сповіщень.
* `VAPID_SUBJECT` — контактний e-mail для VAPID (наприклад, `mailto:admin@domain.com`).
* `INTERNAL_API_KEY` — внутрішній ключ зв'язку між БД та Edge Function (генерується випадково на продакшені).
* `SUPABASE_URL` — адреса API Supabase.
* `SUPABASE_SERVICE_ROLE_KEY` — сервісний секретний ключ Supabase.
